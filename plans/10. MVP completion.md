# Phase 10 MVP Completion

## Summary

- This phase will close only the remaining Must-Have functional gaps in the current repo:
    - real Post concern compose-and-submit flow
    - doc-compliant blocked moderation UX for concern submit, response submit, and feedback comment save
    - feedback comment moderation through one authoritative service-role feedback write path
    - dual notification emission for first-time liked + comment feedback saves
    - minimum hardening so authenticated clients can no longer write public.response_feedback directly
- This phase will not edit docs/, request a new plans/ file, broaden into Phase 11/12, or add draft persistence beyond the current screen
session.

## Implementation Changes

### Concern posting flow

- Files to change: app/post-concern/index.tsx, src/features/concerns/api.ts, new src/features/concerns/api.test.ts.
- app/post-concern/index.tsx will stop being a placeholder and become the real compose screen.
- State will be local React state inside app/post-concern/index.tsx only.
- Draft initialization is intentionally local-only and empty on first entry.
- Draft restore across route leave/re-entry is intentionally out of scope for Phase 10.
- Draft restore across app relaunch is intentionally out of scope for Phase 10.
- Draft survives blocked moderation in the same mounted screen session.
- Draft survives retryable failures in the same mounted screen session.
- Draft is cleared only after an approved submit response.
- Duplicate submits are prevented by an isSubmitting guard.
- While loading, the input is non-editable and the submit button is disabled.
- On approved submit, clear local draft/error state and router.push("/post-concern/my-concerns/[concernId]").
- If the user later comes back to Post concern, they see a fresh empty draft because persistence across re-entry is intentionally not
implemented in this phase.

### Deterministic blocked-UX rules

- Files to change: app/post-concern/index.tsx, app/inbox/[deliveryId].tsx, app/post-concern/my-concerns/responses/[responseId].tsx.
- Concern submit shows the documented warning only for edge result status = "blocked" with code = "moderation_blocked".
- Response submit shows the documented warning only for edge result status = "blocked" with code = "moderation_blocked".
- Feedback save shows the documented warning only for edge result resultCode = "comment_blocked".
- The warning UX is fixed across all three flows:
    - Alert.alert
    - message 부적절한 표현이 감지되었습니다.
    - one action only: 확인
- Concern submit keeps inline validation text only for empty/invalid/too-long body.
- Response submit keeps inline validation text only for invalid delivery id, empty body, too-short body, too-long body.
- Feedback save keeps inline informational/error text only for no_op, example_concern_not_allowed, response_not_accessible, and generic
retryable failures.
- Auth/profile mismatch, moderation unavailable, generic server failure, and network failure remain retryable generic inline failures.
- No other blocked-message presentation style will be introduced in this phase.

### Feedback moderation write path

- There will be one authoritative moderation-aware service-role feedback write path for this flow after Phase 10.
- Implementation may either replace the current RPC or introduce a narrowly compatible successor, but there will be no split-brain end
state.
- All same-phase callers will be migrated together in the same phase:
    - supabase/functions/save-response-feedback/index.ts
    - supabase/functions/save-response-feedback/handler.ts
    - src/features/my-concern-responses/api.ts
    - app/post-concern/my-concerns/responses/[responseId].tsx
- After the migration, those callers will all use the same moderation-aware path only.
- Client and edge callers will continue to expect the same row-oriented notification result normalization pattern:
    - feedback_id
    - result_code
    - notification_id
    - notification_profile_id
    - notification_type
    - notification_related_entity_type
    - notification_related_entity_id
- Result codes after Phase 10:
    - saved
    - no_op
    - example_concern_not_allowed
    - response_not_accessible
    - comment_blocked
- Edge response after update:
    - 200 { resultCode: "saved" | "no_op" | "example_concern_not_allowed" }
    - 200 { resultCode: "comment_blocked", userMessage: "부적절한 표현이 감지되었습니다." }
    - 404 { code: "response_not_accessible", userMessage: ... }
    - generic 401/409/500 unchanged
- Multiple notification rows are already supported in supabase/functions/save-response-feedback/handler.ts through rows.flatMap(...).
- Multiple notification rows are already verified in supabase/functions/save-response-feedback/handler.test.ts.
- The current bug is SQL-only: the write path returns early after the first notification row. Phase 10 will remove that limitation so a
first approved liked + comment save can return two notification rows.

### Feedback moderation semantics

- Files to change: new migration, supabase/functions/save-response-feedback/index.ts, supabase/functions/save-response-feedback/
handler.ts, src/features/my-concern-responses/api.ts, app/post-concern/my-concerns/responses/[responseId].tsx.
- Comment normalization rule is fixed: nullif(btrim(coalesce(commentBody, '')), '').
- Blank comment becomes null.
- Unchanged comment means normalized incoming comment equals normalized stored comment.
- If normalized comment is unchanged and liked is unchanged, return no_op.
- no_op means no moderation call, no response_feedback write, no audit write, no notifications.
- Liked-only first save means no moderation call and normal feedback save.
- Comment-only first save means moderation is required.
- Liked + comment first save means moderation is required and, if approved, two notification rows are returned.
- Blocked changed comment means the write path returns comment_blocked, writes only a moderation audit row with subject_type =
'response_feedback_comment', writes no response_feedback row, and emits no notifications.
- If the request includes both a liked-state change and a changed non-empty comment that blocks, the whole request is blocked and no part
is persisted.
- This full-request atomicity is a deliberate implementation decision, not an explicit docs sentence.
- The docs explicitly require blocked warning UX, draft preservation, and retry after correction.
- The atomic whole-request rule is chosen to preserve deterministic blocked-draft correction behavior and avoid partial-save ambiguity in
a single feedback submit.

### response_feedback hardening

- SQL changes in the new migration will be:
    - revoke insert on public.response_feedback from authenticated;
    - revoke update on public.response_feedback from authenticated;
    - drop policy if exists response_feedback_insert_concern_author_only on public.response_feedback;
    - drop policy if exists response_feedback_update_concern_author_only on public.response_feedback;
- response_feedback_select_concern_author_only stays in place unchanged.
- After this migration, authenticated clients keep read access through RLS but have no direct write path.
- Feedback writes become edge/service-role only in this phase.
- No broader write-path hardening sweep will be done for other tables.

## Migration Verification Method

- Verification will use one repo-local Supabase method only:
    - apply the migration by running npx -y supabase db reset --local
    - run targeted SQL verification against the local Supabase Postgres started by this repo, using docker exec against the repo’s local
    Supabase DB container
- The DB container will be resolved automatically at runtime by a deterministic repo-local method:
    - query docker ps for the running local Supabase DB container created by this repo’s supabase stack
    - use that resolved container in the verification command
    - no hard-coded container name and no ad hoc manual lookup
- The targeted migration verification will check exactly these items for the new feedback migration:
    - direct grants on public.response_feedback no longer allow authenticated INSERT or UPDATE
    - only the select policy remains on public.response_feedback
    - blocked changed comment returns comment_blocked
    - blocked changed comment creates no public.response_feedback write
    - blocked changed comment creates exactly one moderation audit row with subject_type = 'response_feedback_comment'
    - unchanged comment returns no_op without creating a new audit row
    - blank comment saves as NULL
    - first save with liked = true and approved non-empty comment returns two notification rows
    - direct authenticated INSERT and UPDATE attempts against public.response_feedback fail after the migration
- The targeted SQL verification will use minimal fixture rows for:
    - one real concern author
    - one responder
    - one real concern, delivery, and response
    - one example concern, delivery, and response only if needed for a preserved example-denial assertion
- This verification stays narrow to the feedback migration and does not expand into a Phase 11 harness.
