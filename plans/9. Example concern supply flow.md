이 단계에서 해야 할 일은 TODO.md에 이미 남은 example concern 관련 미완료를 닫는 것이다.

해야 할 동작:

Inbox에서 실제 고민이 충분하지 않을 때 example concern을 공급한다. 현재 정보 구조와 화면 골격 섹션에 “내게 전달된 실제 고민 목록 / 실제 고민이 충분하지 않을 때 예제 고민 노출”이 아직 미체크다.
example concern을 UI상 실제 사용자 고민처럼 보이게 처리하되, 내부적으로는 source_type = example로 유지한다. 이 요구는 상단 범위 정의에도 있고, 화면 골격 섹션에도 별도로 미체크 상태다.
example concern의 공급 규칙을 명확히 분리해야 한다. TODO.md 상단 계약은 example concern이 실제 사용자 고민처럼 보이되 내부적으로 태깅되어야 하고, 답변만 가능하며 좋아요/후기/푸시/solved-count/기본 분석에서 제외되어야 한다고 고정한다. 이 계약이 아직 전부 닫힌 것은 아니다.
fallback 정책을 명확히 해야 한다. real concern 라우팅 실패 시 no-delivery를 허용하는 규칙과, example concern 공급을 사용자의 Inbox 경험 보완용으로 넣는 규칙이 섞이면 안 된다. TODO.md는 적격 답변자가 0명일 때만 no-delivery를 허용한다고 박아두고 있어서, example concern을 real concern 라우팅 실패의 대체물처럼 섞는 구현은 위험하다.

즉, phase 9의 본질은 “예제 고민을 넣는다”가 아니라, real concern 흐름과 혼동되지 않도록 공급·노출·차단 규칙을 분리해서 닫는 것이다.
