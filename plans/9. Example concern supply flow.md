## 1. Remaining risky points addressed

- concern_deliveries reuse is acceptable only because origin remains fully derivable from the joined concern row:
    - concerns.source_type = 'real' means routing-created real delivery semantics
    - concerns.source_type = 'example' means Inbox-supplied example delivery semantics
- No extra discriminator is required if this invariant stays true:
    - real routing code never inserts deliveries for example concerns
    - the new supply RPC never inserts deliveries for real concerns
- That makes concerns.source_type sufficient for:
    - Inbox reads: list/detail can branch safely after joining concerns
    - response submission: existing response write path already joins concerns and suppresses notifications for example
    - operational/debug interpretation: any delivery can be classified by joining concerns and inspecting source_type and example_key
    - future solved-count / analytics work: those queries must already join from responses -> concern_deliveries -> concerns; excluding
    example remains a single source-type filter, not a second architecture
- No additional DB origin flag will be added in this phase because it would duplicate information already guaranteed by
concerns.source_type and widen scope without removing a real ambiguity.

## 2. Revised persistence semantics

- Minimum new DB objects:
    - one migration that seeds deterministic example rows into public.concerns
    - one security-definer RPC: public.supply_inbox_example_deliveries(p_target_count integer default 3)
- No new tables.
- Reused existing tables:
    - public.concern_deliveries stores both real and example active Inbox items
    - public.responses stores answers for both real and example deliveries
- Why this is safe:
    - concern_deliveries already models the only state Phase 9 needs for supplied examples: assigned, opened, responded, plus recipient
    ownership and timestamps
    - responses already models the only answer contract Phase 9 needs: one answer per Inbox item via UNIQUE (delivery_id)
    - example-vs-real behavior already branches correctly through the joined concern:
        - notifications/push: suppressed when concerns.source_type = 'example'
        - feedback write/read: blocked when concerns.source_type = 'example'
- Operational meaning of an example delivery row:
    - it is not a routing result
    - it is a persisted Inbox supplement instance for one recipient and one seeded example concern
    - that meaning is derived from concerns.source_type = 'example', not from a second storage model

## 3. Revised supply RPC contract

- RPC name: public.supply_inbox_example_deliveries(p_target_count integer default 3)
- Allowed behavior:
    - may insert rows only for seeded concerns where source_type = 'example'
    - may insert rows only for auth.uid()
    - may insert rows only when the caller’s active example count is below the desired example count
- Desired example count:
    - desired_example_count = max(p_target_count - active_real_count, 0)
    - active_real_count counts caller-owned real deliveries with status assigned or opened
    - active_example_count counts caller-owned example deliveries with status assigned or opened
    - insert count = max(desired_example_count - active_example_count, 0)
- Duplicate prevention:
    - use existing UNIQUE (concern_id, recipient_profile_id) on concern_deliveries
    - only select example concerns the caller has never received in any state
    - use INSERT ... ON CONFLICT DO NOTHING so repeated refreshes or concurrent calls do not create extras
- Repeated calls:
    - once the caller already has enough active example deliveries to satisfy the current gap, the RPC performs no inserts
- Responded examples:
    - never re-supplied to the same user
    - a previously responded (example concern, recipient) pair is considered permanently consumed
- Opened but unanswered examples:
    - remain active
    - remain listable
    - count toward the active example gap on future loads
- Pool exhaustion:
    - yes, the example pool is permanently exhaustible per user
    - once the user has consumed or is already holding all seeded examples, later calls may return fewer than the target count

## 4. Revised example lifecycle rules

- Example deliveries use the existing concern_deliveries.status lifecycle unchanged:
    - assigned: supplied into Inbox, not yet opened
    - opened: opened from Inbox, not yet answered
    - responded: answered, no longer active
- Inbox composition visibility:
    - visible real items: caller-owned real deliveries in assigned or opened
    - visible example items: caller-owned example deliveries in assigned or opened
    - final list order: all visible real items first in existing order, then all visible example items in deterministic example_key ASC
    order
- Supply consumption rules:
    - assigned and opened are active and reusable for future Inbox loads
    - responded is consumed and never reintroduced
- Important nuance:
    - supply insertion is driven by the real-item gap, but already-active example deliveries stay visible until answered
    - this means a user can temporarily see more than the target count if new real deliveries arrive while older example deliveries are
    still active
    - no new examples are inserted in that case

## 5. Revised server-side feedback exclusion explanation

- No new server-side feedback guard is required in this phase because the repository already blocks example feedback in three places:
    - DB trigger private.enforce_real_concern_feedback() rejects any response_feedback row whose joined concern is not real
    - RPC public.save_response_feedback_with_notifications(...) returns example_concern_not_allowed when the joined concern is example
    - RPC public.get_my_response_feedback_for_delivery(...) only returns rows when c.source_type = 'real'
- Phase 9 preserves that contract by:
    - reusing the existing responses model for example answers
    - keeping server-side joins on concerns.source_type
    - additionally skipping feedback fetch/UI for example items on the client only as a UX optimization, not as the primary guard

## 9. Why the plan is now operationally unambiguous

- There is exactly one new mutation surface: supply_inbox_example_deliveries().
- Its insert conditions, duplicate handling, active-state reuse, permanent consumption, and exhaustion behavior are explicitly defined.
- Example items do not create a second response architecture.
- Real routing still owns real delivery vs no-delivery, while Inbox supply only persists recipient-owned example instances after routing
is already finished.
- Every delivery and response remains classifiable by one joined fact: concerns.source_type.
