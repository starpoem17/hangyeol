## 4. Exact test harness design

- Add supabase/tests/harness.ts with:
    - loadLocalSupabaseEnv()
        - reads PHASE11_SUPABASE_URL
        - reads PHASE11_SUPABASE_ANON_KEY
        - reads PHASE11_SUPABASE_SERVICE_ROLE_KEY
        - throws a clear error if any are missing
    - createAnonClient()
        - returns a SupabaseClient configured with local URL + anon key
    - createServiceClient()
        - returns a SupabaseClient configured with local URL + service-role key
    - signInAnonymousUser()
        - creates a fresh anon client
        - calls auth.signInAnonymously()
        - returns { client, user, session }
    - fetchBootstrappedProfile(client, session)
        - calls fetchOwnProfileWithRetry({ supabase: client, session, isCurrent: () => true, delaysMs: [0] })
    - completeUserOnboarding(client, input)
        - calls completeOnboarding(client, input)
    - seed helpers using service client for deterministic setup:
        - insertRealConcern
        - insertExampleConcern
        - insertDelivery
        - insertResponse
        - insertNotification
        - insertPushToken
    - query helpers:
        - listAuditRowsBySubject
        - getConcernById
        - getResponseByDeliveryId
        - getFeedbackByResponseId
        - listNotificationsByProfileId
    - unique seed strategy:
        - every created body/example key/token uses a UUID suffix
    - dependency builders:
        - buildSubmitConcernDeps(serviceClient, options)
        - buildSubmitResponseDeps(serviceClient, options)
        - buildSaveFeedbackHandlerDeps(serviceClient, options)
- Isolation strategy:
    - suite runner performs one supabase db reset --local --yes --no-seed before test execution
    - each test still uses unique users and unique seeded row values
    - each assertion filters by seeded ids, not global counts
    - test order independence is preserved by avoiding shared mutable fixtures

## 9. Risks/blockers

- Local Supabase must be available on Docker; otherwise Phase 11 DB/RLS/RPC verification is blocked.
- The local edge runtime is not a dependable boundary here, so tests intentionally call:
    - submitConcernWithDependencies
    - submitResponseWithDependencies
    - handleSaveResponseFeedbackRequest
    with real RPC-backed persistence beneath them.
- private schema is not exposed in PostgREST, so the “audit blocked” assertion must check schema-access failure, not empty data.
- If current local migrations differ from the pinned repo state at runtime, reset output may reveal drift and block honest closure of TODO
items.
